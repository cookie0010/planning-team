## 설명 방식 (강한 취향)
**개념을 먼저 정의하고, 구체 예시를 붙인 뒤, 비약 없이 단계로 설명.** 전문용어·추상 제안을 앞세우면 안 됨.
**Why:** IA 논의 중 '화면 지도', '지휘석 통합' 같은 표현을 정의 없이 던져 사용자가 두 번 연속 "무슨 말인지 모르겠다"고 함.
**How to apply:** 새 용어는 "이건 ~입니다(예: 1반·2반 캔버스)" 식으로 한 줄 정의+예시 먼저. A→C로 건너뛰지 말고 B를 채울 것. 일상어로.

## 구조는 채팅이 아니라 문서로
화면 목록·흐름 같은 구조는 채팅에 길게 늘어놓으면 사용자가 잡기 어려워함. 사용자가 직접 "ux 안에 넣어주세요"라고 문서화를 요구함.
**How to apply:** IA는 `ia/ia-v<N>.md`로 먼저 쓰고, 채팅엔 짧은 요약 + 파일 경로 + "읽어보고 어디가 어색한지 짚어달라". 긴 표를 채팅에 도배하지 말 것.

## 큰 결정 폼 금지
AskUserQuestion에 3문항(진입점·첫동선·템플릿)을 한 번에 던졌다가 거부당함.
**How to apply:** 한 번에 결정 여러 개를 묻지 말 것. 합리적 기본값을 제안+문서에 '아직 안 정한 것'으로 플래그하거나, 정말 필요한 **한 가지만** 콕 집어 물을 것.

## 버전·정합 규칙
- 산출물 버전 분리: 내용 개정 = `-v<N>` 새 파일, 워딩만 = `-v<N>.1`. 덮어쓰기 절대 금지(one-pager·PRD·IA 공통).
- **PRD = source of record.** 상류 one-pager가 어긋나도 다시 맞추지 않음(재공유 필요할 때만 개정). IA는 PRD를 따른다.
- 워딩: 군더더기·현학 라벨·§ 기호 금지, 정직 프레이밍, '해결책' 단정 금지.

## 벤치마크는 웹검색 말고 캡처로
경쟁 도구의 UI·화면 구조는 텍스트/웹검색으로 거의 안 잡힘. 사용자가 직접 찍어 `reference/`에 넣어준 캡처가 진짜 입력이었음.
**How to apply:** UI 구조 벤치마크가 필요하면 deep-research 돌리기 전에 사용자에게 필요한 화면 캡처를 콕 집어 요청. deep-research 범위도 착수 전 좁게 확정(이번에 대상이 두 번 잘림: Padlet 제외, 4종으로). 텍스트로 알 수 있는 건 동선 단계 수·옵션 이름 정도(공식 도움말 문서).
