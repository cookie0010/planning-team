# IA (정보 구조): quizn-board-v2

> Project: `quizn-board-v2` / Date: 2026-06-16 / Version: v6
> 기준 문서: `prd-v9.md`(권위 있는 스펙) + `ux/ia-v5.md`(직전 IA) · HTML 와이어프레임 `ux/wireframe/`(t-a-list-v2·t-a-group-v2·t-template) · 벤치마크 `ux/benchmark-v1.md`
> 다음 단계: 나머지 화면(B·C·D·E·F·G)도 HTML로 잡으며 IA에 반영 → 흐름도 갱신 → 검증 게이트

> **v5→v6 바뀐 점(요약):** 와이어프레임을 Figma 대신 **HTML로 다시 잡으며**(A 캔버스 목록부터) 나온 **A 화면의 구조 결정**을 반영했다. 핵심은 **그룹의 위상 승격** — v5에서 그룹은 "A 위에서 일어나는 정리 동작(새 화면 아님)"이었는데, v6에서는 **A의 드릴인 하위 뷰(A-그룹 내부)**가 된다. ① **캔버스가 주연, 그룹은 폴더.** A에서 캔버스는 라이브 썸네일 카드로 직접 노출되고, 그룹은 Figma 프로젝트 폴더처럼 '미니 썸네일 2×2 + 그룹명 + 캔버스 개수'의 폴더 카드로 같은 그리드에 섞여 표시된다. ② **그룹 폴더는 클릭해 들어가는 네비게이션 대상.** 폴더를 누르면 그 그룹의 캔버스 목록(A-그룹 내부)으로 진입하고, 거기서 캔버스를 연다(→B). 브레드크럼으로 A 복귀. ③ **생성 동선 명시** — A 헤더에 '캔버스 생성'(→T)과 '그룹 만들기' 두 동작, 그리드 맨 앞에 '새 그룹 만들기' 타일, 그룹 내부에는 '이 그룹에 캔버스 추가'. ④ **캔버스 카드 단위 동작** — 더보기(⋮) 메뉴로 보관·삭제, 잠금 카드는 복제. 잠금은 **카드 상태 배지일 뿐 목록 필터가 아니다**(상태 enum은 v5대로 유지). ⑤ **'지휘석'은 A의 개념적 역할로만 유지하고 UI 라벨로는 노출하지 않는다** — 드릴다운은 카드의 '크게 보기'로 표현. **나머지 화면(B·C·D·E·F·G)은 v5 내용을 그대로 유지**하되, 위 A 결정과 닿는 부분만 §2 연결도에서 정리했다. 계층·역할·뷰어 정원 포함·G 뷰어 재진입 등은 v5와 동일.

## 0. 이 문서가 무엇인가

이 제품에 **화면이 몇 개 있고, 각 화면이 무엇을 보여주고, 화면끼리 어떻게 이어지는지**를 적은 설계도다. 실제 그림(와이어프레임)을 그리기 전에, 그릴 대상의 목록과 연결을 먼저 확정하는 단계다. 이번 버전은 그 와이어프레임을 HTML로 그리다 A 화면에서 나온 구조 결정을 거꾸로 IA에 반영한 것이다.

전체는 두 area로 나뉜다 — **교사가 쓰는 화면**(데스크톱)과 **학생이 쓰는 화면**(태블릿·폰). 둘이 만나는 유일한 접점은 캔버스다.

- 교사 화면 기준: 데스크톱 1440px
- 학생 화면 기준: 태블릿 1024px (+ 폰 390px 변형)

### 먼저 짚는 용어: 역할 4단계

화면을 보기 전에 한 줄씩 정의한다. PRD v9 §1을 그대로 따른다.

- **소유자** = 프로젝트를 만든 사람(보통 교사). 로그인 필수. 모든 것을 제어한다.
- **편집자** = 소유자나 다른 편집자가 임명·초대한 사람. 로그인 필수. 임명된 캔버스의 페이지 추가·템플릿 설정·편집이 가능하다.
- **참여자** = 캔버스에 들어와 **함께 만드는** 사람(보통 학생). 무계정 입장 가능. 객체 생성·이동·편집 + 포스트잇 공감 투표.
- **뷰어** = 캔버스에 들어와 **보기만** 하는 사람. 무계정 입장 가능. 객체 편집 불가, 페이지 변경 불가, 실시간 반영을 보기만. 공감 투표 가능 여부는 미정(기본 불가) — PRD OQ-21.

> 화면 수준에서 핵심은: 소유자·편집자는 **교사 area**(데스크톱)에 산다. 참여자·뷰어는 **학생 area**(태블릿·폰)에 산다. 참여자와 뷰어는 **같은 화면(E)**을 쓰되, 뷰어는 그 화면의 '보기 전용' 변형으로 들어온다.

### 계층 (PRD v9 §2 그대로)

위에서 아래로 담는 관계다.

```
프로젝트 (최상위 통 — 소유자가 생성. PoC에선 사실상 1개. 다중 프로젝트 화면은 범위 밖, 개념상으로만 위에 존재)
 └ 그룹 (선택 — 프로젝트 안에서 캔버스끼리 묶는 단위. 예: "1차시", "모둠 발표". 필수가 아니라 안 묶어도 됨)
    └ 캔버스 (권한별 QR · 학생 진입 단위 · 진입자 20명(편집자·참여자·뷰어 모두 포함) · 최대 30페이지)
       └ 페이지 (편집자가 템플릿을 붙여 구성 · 최대 30개 · 좌우 추가·이동)
          └ 객체 (텍스트·그리기·포스트잇·도형)
```

> v5와 같은 계층이다. v6도 이 계층 자체는 건드리지 않는다. **다만 화면 매핑이 달라진다** — v5에서 '그룹'은 화면을 만들지 않는 정리 동작이었는데, v6에서는 이 계층의 '그룹' 층이 **A의 한 단계 안쪽 화면(A-그룹 내부)**으로 드러난다. 즉 계층은 그대로지만, 그 중 그룹 층이 이제 클릭으로 드나드는 네비게이션 단계가 됐다.

## 1. 화면 목록

### 교사 화면 (4개 + A의 드릴인 하위 뷰 1종)

| 코드 | 화면 | 무엇을 보여주나 | 무엇을 하나 | PRD 기능 |
|---|---|---|---|---|
| **A** | 캔버스 목록 | 이 프로젝트의 **캔버스 카드(라이브 썸네일)**들과 **그룹 폴더 카드**가 한 그리드에 섞여 보임. 캔버스 카드 = 상태 배지·참여 인원·객체 수·페이지 수, 폴더 카드 = 미니 썸네일 2×2 + 그룹명 + 캔버스 개수 | **[+ 캔버스 생성]**(→T), **[그룹 만들기]**, 상태 칩(전체/진행 중/보관)으로 보기 전환, 캔버스 카드의 '열기'(→B)·'크게 보기'(드릴다운)·더보기(⋮ 보관·삭제)·잠금 카드 '복제', **그룹 폴더 클릭 → A-그룹 내부** | F1·F2·F7·F11 |
| **A-그룹 내부** | 한 그룹의 캔버스 목록 | 그 그룹에 속한 캔버스 카드들만(라이브 썸네일), 상단에 그룹명·캔버스 개수·브레드크럼 | 그룹 안 캔버스 '열기'(→B)·'크게 보기'(드릴다운), **[+ 이 그룹에 캔버스]**, 그룹 이름 변경, **브레드크럼으로 A 복귀** | F11·F7·F1 |
| **T** | 템플릿 선택 | 선택지 — **빈 캔버스 + 템플릿 3종**(브레인스토밍·마인드맵 등, 개수는 OQ-03), 각 미리보기 | 하나 고르면 그 템플릿이 깔린 새 캔버스가 만들어지고 즉시 B로 진입 | F3·F1 |
| **B** | 캔버스 편집 (편집자 뷰) | 페이지(좌우로 넘김, N/30 표시), 도구바, 잔여 시간, 객체 작성자 | 객체 만들기·편집, 페이지 추가·템플릿 설정(최대 30), QR 공유, 타이머·잠금 | F4·F3·F8 |
| **C** | 캔버스 설정 | 편집자 목록, 익명/닉네임·비번 옵션 상태 | 편집자 임명(캔버스 단위), 옵션 on/off | F6 |

**A와 A-그룹 내부의 관계 (v6 핵심):**

- 둘은 **같은 종류의 화면**이다 — 캔버스 카드 그리드. 차이는 **무엇을 담느냐**뿐이다. A는 '그룹에 안 묶인 캔버스 + 그룹 폴더'를 섞어 보이고, A-그룹 내부는 '그 그룹에 묶인 캔버스만' 보인다.
- 그래서 A-그룹 내부는 **새로운 종류의 화면이 아니라 A의 한 단계 안쪽(드릴인) 뷰**다. 별도 코드를 주되(연결을 또렷이 그리려고), 와이어프레임에서는 A 그리드를 재사용하는 같은 레이아웃으로 본다.
- **드나드는 법:** A의 그룹 폴더 카드를 클릭 → A-그룹 내부로 들어감. 상단 **브레드크럼('← 캔버스 / 그룹명')으로 A 복귀**.

**별도 화면이 아니라 위에 겹쳐 뜨는 것들:**

- **템플릿 선택 모달** — A에서 [+ 캔버스 생성]을 누르거나, 퀴즈앤 링크로 처음 진입하면 자동으로 뜬다. 내용은 위 **T 화면과 같다**(같은 선택 흐름). → 목록에는 'T'로 한 번만 올리고, 자동/수동은 진입 경로 차이로만 본다. (A-그룹 내부의 [+ 이 그룹에 캔버스]도 같은 T를 띄우되, 만든 캔버스가 그 그룹에 들어간다.)
- **그룹 만들기 동작** — A 헤더의 [그룹 만들기] 또는 그리드 맨 앞 **'새 그룹 만들기' 타일**로 시작한다. 새 그룹을 만들고 캔버스를 담는 인터랙션(드래그? 다중 선택 후 '그룹으로'?)은 아직 안 정함(§4). v5에서는 이걸 'A 위 정리 동작'으로만 봤는데, v6에서는 **그 결과물인 그룹이 폴더 카드 + A-그룹 내부라는 화면으로 드러난다** — 만드는 동작은 여전히 A 위 오버레이/인라인이고, 만들어진 그룹은 네비게이션 대상이 된다.
- **QR 공유 패널** — B 안에서 [공유] 버튼으로 띄움(교실 프로젝터용 큰 화면). **공유할 권한을 먼저 고른다 — 편집자 / 참여자 / 뷰어.** 고른 권한에 해당하는 QR·URL을 발급·표시한다. 권한마다 별개 링크다. 이 권한 선택이 학생 입장 화면(D)의 분기를 결정한다.
- **캔버스 카드 더보기(⋮) 메뉴** — A·A-그룹 내부의 각 캔버스 카드에 달린 메뉴. **보관 / 삭제**, 그리고 (잠금·보관 카드의 경우) **복제**. 카드 위에 잠깐 겹쳐 뜨는 작은 메뉴이지 화면이 아니다.

> **상태와 필터는 다르다(v6 명확화).** 캔버스 상태 enum은 v5대로 **진행 중 / 잠금(타이머 만료) / 보관** 셋이다. 이 중 **잠금은 카드 위 상태 배지로만 표시**한다 — 목록을 거르는 필터가 아니다. 목록 상단의 상태 칩은 **전체 / 진행 중 / 보관** 두 가지 보기 전환만 둔다(잠금은 진행 중과 함께 보이는 게 자연스러워서, 잠금 전용 필터 칩은 두지 않음). 즉 **상태 3종(데이터)** ≠ **필터 칩 2종(보기)**.

### 학생 화면 (4개 화면 + E의 권한 변형 2종)

> 학생 area는 v5에서 바뀐 것이 없다. 아래는 v5 그대로다(요약). A의 구조 변경은 교사 area 안에서만 일어나며 학생 동선에 영향이 없다.

| 코드 | 화면 | 무엇을 보여주나 | 무엇을 하나 | PRD 기능 |

| **D** | 입장 화면 | 캔버스 이름. **들어온 QR의 권한에 따라 입장 방식이 갈림** | 권한별 분기에 따라 로그인 또는 닉네임+4글자 비번 입력 → 입장 | F5 |
| **E** | 캔버스 편집 | 편집자가 짜둔 페이지, 하단 도구바, 잔여 시간, presence | **권한 변형 2종**(아래) | F4·F10 |
| **F** | 안내 화면 | 상황 메시지 | 없음(읽고 대기) — 정원 초과 / 잠금(보기 전용) / 보관, 3종 변형 | F5·F8 |
| **G** | 내 활동 (로그인 시) | 참여했던 캔버스 목록 | **재진입 — 그때의 권한대로 E-참여자 또는 E-뷰어로** | F5 |

**E의 권한 변형 2종:** E-참여자(객체 작업 + 공감 투표) / E-뷰어(보기 전용, 새 화면 아니라 E의 변형). **D의 권한 분기:** 편집자 QR=로그인 필수→B / 참여자 QR=로그인 or 닉네임+비번→E-참여자 / 뷰어 QR=로그인 or 닉네임+비번→E-뷰어. 정원 20명은 편집자·참여자·뷰어 모두 포함, 초과 시 F. (상세는 v5 §1과 동일 — 변경 없어 여기서는 줄인다.)

## 2. 화면 연결 (네비게이션)

```
교사 area (데스크톱)
  A 캔버스 목록                          ← 이 프로젝트의 landing/return 화면 (개념상 '지휘석')
   │  ┌ 캔버스 카드(라이브 썸네일)  ─[열기]──────▶ B 캔버스 편집
   │  │                          └[크게 보기]──▶ 드릴다운(자기 화면 확대)  ※ broadcast 아님(PRD 비목표 5)
   │  │                          └[⋮]─────────▶ 보관 / 삭제 / (잠금·보관 카드)복제
   │  └ 그룹 폴더 카드 ──[클릭]──────────────────▶ A-그룹 내부
   ├─[+ 캔버스 생성] ───────────────▶ T 템플릿 선택 ──[고름]──▶ B 캔버스 편집
   ├─(자동) 템플릿 선택 모달 ────────▶ (= T 내용) ──[고름]──▶ B 캔버스 편집   ← 링크 첫 진입 시
   ├─[그룹 만들기] / '새 그룹' 타일 ─▶ 새 그룹 생성(담기 동작, A 위 인라인) → 그룹 폴더 카드로 나타남
   └─[설정] ─────────────────────────▶ C 캔버스 설정

  A-그룹 내부 (한 그룹의 캔버스 목록 = A의 드릴인 하위 뷰)
   ├─[← 캔버스] 브레드크럼 ──────────▶ A 복귀
   ├─ 캔버스 카드 ─[열기]────────────▶ B 캔버스 편집
   │              └[크게 보기]──────▶ 드릴다운(이 그룹 단위)
   └─[+ 이 그룹에 캔버스] ───────────▶ T 템플릿 선택 ──[고름]──▶ B (그 그룹에 소속)

  B 캔버스 편집
   └─[공유] ─── QR 공유 패널 ─── 권한 선택(편집자 / 참여자 / 뷰어)
                                  ├─ 편집자 QR·URL ┐
                                  ├─ 참여자 QR·URL ┤  (권한별 별개 링크, 학생에게 전달)
                                  └─ 뷰어   QR·URL ┘

학생 area (태블릿·폰)  ※ v5와 동일
  편집자 QR ──▶ D 입장(로그인 필수) ──[로그인]──▶ B 캔버스 편집(교사 area로 합류)
  참여자 QR ──▶ D 입장(로그인 or 닉네임+4글자 비번) ──▶ E-참여자
  뷰어   QR ──▶ D 입장(로그인 or 닉네임+4글자 비번) ──▶ E-뷰어(보기 전용)
                         └─(정원초과·잠금·보관) ──▶ F 안내 화면   ※ 정원 20명은 편집자+참여자+뷰어 모두 포함
  로그인 학생: G 내 활동 ──▶ E-참여자 재진입  (참여자로 참여했던 캔버스)
                        └──▶ E-뷰어   재진입  (뷰어로 봤던 캔버스)
```

두 area의 연결고리는 **B의 QR 공유 패널에서 발급한 권한별 QR**이다. 그중 **편집자 QR은 D를 거쳐 다시 교사 area(B)로 합류**하고, 참여자·뷰어 QR은 학생 area(E)에 머문다. **G(내 활동)는 학생 area 안에서의 재진입 통로**다.

> **v6에서 달라진 곳은 A의 안쪽뿐이다.** v5에서 A는 평평한 한 화면(그룹은 그 위 정리 동작)이었지만, v6에서는 **A → A-그룹 내부**라는 한 단계 드릴인이 생겼다. 둘 다 결국 캔버스 카드에서 B(열기) 또는 드릴다운(크게 보기)으로 빠진다 — 도착지는 v5와 같고, 가는 경로가 그룹을 거칠 수 있게 됐다.

> 맨 위에 '프로젝트'가 한 층 더 있지만(여러 프로젝트 전환), PoC에선 프로젝트 1개만 쓰므로 그 화면은 그리지 않는다. A(캔버스 목록)가 곧 그 단일 프로젝트의 안쪽이다.

## 3. 핵심 흐름

**① 교사 첫 사용 (실측 2액션)**
```
퀴즈앤 링크 클릭 → A + 템플릿 선택 모달 자동 → [템플릿 선택] → B(캔버스) → [공유 버튼] → 권한 고르고 QR
```
PRD의 "3액션·30초" 기준 충족(2액션 = 템플릿 선택 1 + 공유 1). 권한 선택은 QR 패널 안에서 일어나므로 별도 액션으로 세지 않되, 와이어프레임에서 '기본 참여자 권한 미리 선택'으로 2액션을 지킨다(§4). **그룹 만들기는 선택이라 이 첫 사용 흐름에 들어가지 않는다.**

**② 교사 재사용 — 새로 만들기 / 복제 두 갈래**
```
(새로)  A → [+ 캔버스 생성] → T(빈 캔버스/3종 중) → B → 공유(권한)→QR
(복제)  A → 캔버스 카드 [⋮ → 복제] (또는 잠금 카드의 [복제]) → 복제본(B) → 공유(권한)→QR
```
잘 만든 캔버스 하나를 1반·2반·3반용으로 통째 복제한다. 복제본은 각각 독립이고 QR도 따로. 묶어 두고 싶으면 **A에서 그룹으로 묶을 수 있다(선택)** — 묶으면 그 그룹 폴더로 들어가 한곳에서 본다.

**③ 교사 — 그룹으로 정리하고 그 안에서 작업 (v6 신규 흐름)**
```
A → [그룹 만들기] / '새 그룹' 타일 → 캔버스 담기 → 그룹 폴더 카드 생성
A → 그룹 폴더 [클릭] → A-그룹 내부 → 캔버스 [열기]→B / [크게 보기]→드릴다운 / [+ 이 그룹에 캔버스]→T
A-그룹 내부 → [← 캔버스] 브레드크럼 → A 복귀
```
"1차시", "모둠 발표"처럼 캔버스를 묶어 두고, 폴더를 눌러 그 그룹의 캔버스만 모아 본다. 이 그룹 단위가 곧 지휘(드릴다운)의 자연스러운 범위가 된다(아래 ⑤, OQ-20).

**④ 학생 입장 — 권한별 3갈래 (v5와 동일)**
```
(참여자) 참여자 QR/URL → D(로그인 or 닉네임+4글자 비번) → E-참여자(함께 만들기)
(뷰어)   뷰어   QR/URL → D(로그인 or 닉네임+4글자 비번) → E-뷰어(보기 전용)
(편집자) 편집자 QR/URL → D(로그인 필수) → B(편집자로 합류)
```
참여자·뷰어는 무설치·무가입, 로딩 3초 이내 목표. **정원 20명은 편집자·참여자·뷰어를 합쳐 센다** — 21번째는 누구든 F(정원 초과 안내)로 빠진다.

**⑤ 학생 재진입 (로그인 학생, v5와 동일)**
```
G(내 활동) → 참여했던 캔버스 고름 → 그때 권한대로 ├─ 참여자였으면 → E-참여자
                                              └─ 뷰어였으면   → E-뷰어(보기 전용)
```

**⑥ 교사 지휘 (수업 중)**
```
A(또는 A-그룹 내부) 라이브 썸네일 → '크게 보기'(드릴다운, →자기 화면 확대) / '열기'(→B) / 타이머 제어
```
캔버스 6개를 한 화면에서 스크롤 없이 파악(PRD F7 품질 기준). 막힌 모둠을 돕고, 한 캔버스를 골라 자기 화면에서 크게 본다. **'지휘석'은 A의 개념적 역할로만 두고 UI 라벨로 쓰지 않는다** — 드릴다운은 카드의 '크게 보기'로 표현. 지휘 대상 범위는 '한 그룹'을 기본으로 두되 확정은 §4(OQ-20). 그룹으로 묶어 두면 A-그룹 내부가 그 지휘 단위가 된다.

## 4. 아직 안 정한 것 (그림 그리며 판단)

이번 버전에서 새로 또렷해진/남은 열린 점:

- **그룹 만들기·담기 인터랙션.** 그룹 폴더가 네비게이션 대상이 된 건 확정. 하지만 **새 그룹을 만들고 캔버스를 담는 동작**(드래그로 폴더에 떨어뜨리기? 캔버스 다중 선택 후 '그룹으로 묶기'? 그룹 만들고 안에서 추가?)은 와이어프레임에서. 이번 IA는 'A 헤더 [그룹 만들기] + 그리드 맨 앞 새 그룹 타일에서 시작 → 폴더 카드로 나타남'까지만 확정.
- **그룹 폴더 카드의 라이브성.** 캔버스 카드는 수업 중 2초마다 갱신되는 라이브 썸네일이다. 그룹 폴더 카드의 미니 썸네일 2×2도 라이브로 갱신할지, 정적 대표 썸네일로 둘지는 와이어프레임에서(성능·인지 부하 함께 판단).
- **잠금 카드의 동작 차이.** 잠금·보관 캔버스 카드는 '열기' 외에 '복제'를 노출한다(HTML 와이어프레임). 진행 중 카드는 '열기/크게 보기'. 카드 상태별 액션 세트(어느 상태에 어떤 버튼)를 와이어프레임에서 확정.
- **A 상태 칩의 범위.** 현재 전체 / 진행 중 / 보관 3칩으로 그렸다(잠금은 배지로만, 필터 칩 없음). 보관 캔버스가 같은 그리드에 섞이는지 별도 보기로 빠지는지를 와이어프레임에서.
- **QR 공유 패널의 권한 선택 UX.** 탭 전환식(한 번에 한 QR, 참여자 기본 선택)으로 그렸다 — 이대로 2액션을 지킬지, 세 QR 동시 표시안을 둘지 확정 필요.
- **D 입장 화면의 분기 표현 / G의 권한 표시 / 뷰어 공감 투표(OQ-21) / 4글자 비밀번호(OQ-11) / F 정원 초과 문구** — 모두 v5 §4 그대로 유지(학생 area는 이번에 안 건드림). 상세는 ia-v5 §4 참조.

이전부터 이어지는 열린 점:

- **지휘석(드릴다운) 범위 (PRD OQ-20).** 프로젝트 안 전체 캔버스를 보나, 한 그룹만 보나? 캔버스 6개 한 화면 기준이면 보통 한 그룹 단위가 자연스럽다. **v6에서 A-그룹 내부 화면이 생기면서 이 화면이 곧 '한 그룹 단위 지휘'의 자연스러운 그릇이 됐다** → '한 그룹 단위'를 기본값으로 깔되 여전히 flag(OQ-20 미해소).
- **복제 시 자동 그룹 묶임 여부 (PRD OQ-19).** 1반·2반·3반 복제본이 자동으로 한 그룹이 될지 수동으로 묶을지 미정 → 일단 수동 가정.
- **템플릿 선택지 개수 (PRD OQ-03).** PRD v9 = 빈 캔버스 + 3종(브레인스토밍·마인드맵 등)이나 OQ-03에서 3종 자체가 검증 중. 흐름엔 '빈 캔버스 + 템플릿들'로 두고 개수는 와이어프레임에서(현재 HTML은 빈 캔버스 + 2종으로 그림).

> **PRD와의 정합 메모:** 이번 v6의 A 결정들은 **PRD v9와 어긋나지 않는다.** PRD v9 §2는 그룹을 "그룹 카드 안에 속한 캔버스 썸네일들이 보이고 그룹명 라벨이 붙는다"로, F11은 "그룹 카드(그룹명 라벨, 소속 캔버스 썸네일들)"로 적어 두어 — **폴더 카드 + 그 안의 캔버스 목록**이라는 v6 해석을 이미 지지한다. v6는 그 '소속 캔버스를 보는 화면'을 A-그룹 내부라는 드릴인 뷰로 명시화했을 뿐, 새 제품 결정을 만들지 않았다. **상위 문서로 돌려보낼 갭은 발견되지 않음.**
> 단 PRD OQ-20(지휘 범위)은 A-그룹 내부의 등장으로 '한 그룹 단위' 쪽 근거가 강해졌으니, 차후 PRD 갱신 시 OQ-20을 이 화면 구조에 맞춰 해소할지 @prd-writer가 판단할 거리다(블로킹 아님).
> (그 외 PRD 잔여 OQ는 prd-v9 §9 참조 — OQ-02 검증규모·OQ-03 템플릿·OQ-11 닉네임/비번·OQ-19 복제 규모·OQ-21 뷰어 투표 등. 와이어프레임 진행에는 막힘 없음.)

## 부록. 흐름도 상태

> **FigJam 흐름도 갱신 필요(미반영).** 이 v6는 백그라운드 작업으로 진행돼 FigJam 인증을 거치지 않았다. 기존 흐름도(`ux/ia-flowchart-v5.png`, fileKey `OT6YlTAIzTNYMrf0m9yRoo`)는 **A를 평평한 한 화면 + 그룹=점선 부가동작**으로 그려 v6와 어긋난다. 다음 대화에서 갱신할 때 반영할 변경점:
> - 교사 area subgraph에 **A-그룹 내부** 노드 추가(A에서 그룹 폴더 클릭 → A-그룹 내부, 브레드크럼으로 A 복귀하는 양방향 엣지).
> - v5의 '그룹 만들기 (선택)' 점선 부가동작은 **'그룹 폴더 → A-그룹 내부' 네비게이션 엣지**로 승격(그룹 만들기 동작 자체는 A 위 인라인으로 라벨만 유지).
> - 캔버스 카드 분기를 **열기(→B) / 크게 보기(드릴다운) / ⋮(보관·삭제·복제)**로 분리 표기.
> - '지휘석' 라벨은 노드명에서 빼고 A의 역할 설명(엣지/주석)으로만.
> - 학생 area는 v5 흐름도 그대로 유지.
